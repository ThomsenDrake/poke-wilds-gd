# Building And Placement

Building is the keystone slice: the later campfire/bed/pen/storage behaviors (Phases 2-3) all depend on a placement system, and it gives the dead-end harvest economy (`log` / `hard_stone` / `dry_soil` from the harvest slice) its FIRST consumers. Structure definitions are pure domain data + rules (`scripts/domain/structures.gd`); placement orchestration lives in `scripts/runtime/build_runtime.gd`; the ghost preview, build-mode input loop, and occupancy queries live in `scripts/runtime/structure_layer.gd`, which COMPOSES `world_view.gd` (never extends it). Placed structures persist on the world's sparse override map (see [bootstrap-and-overworld.md](bootstrap-and-overworld.md) and [harvest-and-mutation.md](harvest-and-mutation.md)). Phase 2 (camping/crafting/night survival) fills the behavior of the two Phase-1 placeholders ON TOP of this system — the campfire becomes a light source + crafting station (with a runtime `lit` toggle carried as an additive field on the placement entry) and the bed becomes a full heal + status cure — and adds the TORCH structure (appended LAST to the id list so every existing cycle index stays stable). Full behavior in [camping-crafting-survival.md](camping-crafting-survival.md); this spec keeps owning the placement/occupancy/demolition contracts those behaviors ride on.

## Supported behavior

- The player places structures on bare or already-cleared tiles: walls, roofs, doors, interior partitions, fences, a campfire pad, a bed, and a storage box. A small, fully enclosed house with a door is buildable from harvested materials (the Phase-1 exit object).
- Placement is tile-based and permanent. A structure occupies one tile; the placed sprite renders through the world view's existing prop pipeline (y-sorted against the player's feet), so a placed tile reads exactly like a world prop.
- Placing requires a clear tile: the harvest→build loop is enforced because a structure can only go on a walkable tile with no prop — cut a tree, then build on the cleared tile, or build on open ground.
- Materials are consumed from the bag through the session item API (`get_item_count` / `remove_item`) at the moment of placement; a placement that cannot be afforded is refused and consumes nothing.
- Capability is party-based and reuses the field-move model: `Build` is a Fighting-type field move (`scripts/domain/field_moves.gd` `AUTO_TYPES["build"] == "FIGHTING"`). The starting Chikorita CANNOT Build — scenarios that need it swap in a Build-capable mon (e.g. MACHOP). The overworld context action and the party-screen `FIELD MOVE` action both gate on a capable party member.
- Refusals are traced, never silent: missing materials, an incapable party member, an occupied/non-placeable tile, a self-enclosing placement (`would_trap`), and the shared override cap each emit a `structure_refused` trace carrying the reason.
- A built tile can always be demolished again (the faithful escape from the build loop — see Demolition below), and a placement can never permanently wall the player in on their own tile (the `would_trap` guard).

### Structures and costs

Structure ids are fixed: `wall`, `roof`, `door`, `partition`, `fence`, `campfire`, `bed`, `storage_box`, and since Phase 2 `torch` — APPENDED LAST (after `bed`), so every existing build-cycle index and every scenario that calls `try_place` by explicit id stays stable. The default (Log+Grass→`log`+`dry_soil`) costs, the campfire/box/bed costs, the torch cost, and the door-between-fences gate are faithful to the original PokeWilds Buildings / House_Building wiki pages, with two documented mappings: the original "Grass" secondary (which the harvest slice does not yield) maps to `dry_soil` (thatch/daub), making building the first consumer of all three headline harvest yields (log, hard stone, dry soil); and Desert shells swap Hard Stone for Log+Grass ("Desert homes require Hard Stone"). The torch cost (1 `log` + 1 `dry_soil`) is the original "1x Log & 1x Grass" under the same Grass→`dry_soil` mapping. Item ids are the lowercase bag ids the harvest slice already uses. Two details are documented ASSUMPTIONS rather than scraped fact: the exact per-structure desert quantities (2 `hard_stone` for wall/roof/door/partition, 1 for fence) and treating the SAND (beach) biome as desert-shell — the wiki states only that desert homes "require Hard Stone" with no per-structure counts and names only "Desert".

| --- | --- | --- | --- |
| `wall` | 1 log + 1 dry_soil | 2 hard_stone | no |
| `roof` | 1 log + 1 dry_soil | 2 hard_stone | no |
| `door` | 1 log + 1 dry_soil | 2 hard_stone | yes |
| `partition` | 1 log + 1 dry_soil | 2 hard_stone | no |
| `fence` | 1 log + 1 dry_soil | 1 hard_stone | no |
| `campfire` | 4 log + 2 dry_soil | 4 log + 2 dry_soil | no |
| `storage_box` | 2 log | 2 log | no |
| `bed` | 4 log + 1 soft_bedding | 4 log + 1 soft_bedding | no |
| `torch` | 1 log + 1 dry_soil | 1 log + 1 dry_soil | no |

Biome appearance (sprite family) swaps per biome (default `house1_*`; SAND/DESERT stone/ruins; snow, savanna, and spooky variants from the `pokewilds/tiles/buildings/` families — see tech debt for the partial mapping). A door placed adjacent to a fence renders as a fence gate (the faithful `fence1gate1` variant). The gate flag is a PLACEMENT-TIME SNAPSHOT — the fence-neighbor check runs at placement and is stored on the placement entry (render reads the stored flag, never re-deriving it); the ghost preview uses the identical check, so preview and placement always agree. A later fence change leaving a door's gate art stale is accepted as cosmetic-only (gate vs door is a sprite choice; walkability, cost and refund are unaffected). The campfire keeps the original "4 Log & 2 Grass" crafting-station shape (universal, not biome-swapped).

### Occupancy

- Solid (block movement, carry a block reason): `wall`, `roof`, `partition`, `fence`, `campfire`, `bed`, `storage_box`, `torch` ("A torch blocks the way."). The original also mounts torches on fences/interior walls; the port's one-structure-per-open-tile model cannot express mounting, so torch placement is ground-only (documented deviation — Phase 3 building interiors).
- Walkable (the only opening in a wall line): `door`. Because the door is the sole walkable tile in a wall line, an enclosed wall loop plus one door yields an interior reachable only through that door — "connected rooms" in Phase 1 is occupancy-only.
- Occupancy agrees across logic, render, and collision BY CONSTRUCTION: placements apply at the single `world_generator.get_tile_logic` boundary where clears already apply (setting the prop sprite, walkability, and block reason), so every reader — traversal logic, the world view render, and the player-avatar collision — sees the same tile. There is no second occupancy source to drift.

### Build mode

- Overworld context `Z` on a walkable, non-harvestable tile with a Build-capable party member enters build mode targeting the faced tile. (Harvest still takes precedence: if a harvestable action exists on the faced tile, it resolves first; build mode only opens on "nothing left here".)
- Party-screen `FIELD MOVE` selecting `build` enforces the SELECTED mon's capability and enters build mode constrained to that mon; an incapable mon shows "X can't use that here." This makes the party-screen `FIELD MOVE` real (it previously ignored the selected move id and routed everything to harvest).
- In build mode, movement keys cycle the selected structure, `Z` (action_a) confirms placement, and `X` (action_b) exits. Player movement is disabled while build mode is active. The original's C/V shoulder-button cycle maps to the movement keys (documented deviation; no new InputMap actions).
- A ghost preview shows the selected structure's sprite at the target tile — normal tint when placeable and affordable, a distinct invalid tint otherwise — positioned with the world view's tile-bottom origin so it sorts against the player's feet.

### Demolition (faithful to the original)

- In the original PokeWilds, Cut refunds ALL materials built into a tile, hard-stone structures (graves, statues) need Smash, and terrain tiles need Dig. Phase 1 implements the first two rules (no Phase-1 structure carries a dig-only cost, so demolition is always Cut or Smash): `structures.demolish_move_for(id, biome)` returns `smash` when `cost_for(id, biome)` carries `hard_stone` (the desert/sand stone shells), else `cut`.
- Demolition joins the harvest Z-action flow rather than inventing a second one: overworld `Z` or party-screen `FIELD MOVE` on a built tile routes through `game_runtime.harvest_tile` to `build_runtime.try_demolish` instead of "There is nothing left here." (which previously left a player who enclosed themselves with NO escape). Capability uses the required move (party-wide, or the constrained mon), with the harvest-style refusal wording ("A built wall blocks the way. It could be CUT."). Since Phase 2 the field-action router reads the FACED structure first: a campfire opens the camp menu (Craft / Extinguish-or-Light / DEMOLISH entries) and a bed rests the party directly — demolition STAYS reachable via the menu entry, so the region-seal escape below is preserved; every other structure keeps this direct path. `placement_flow`/`placement_flow_demolition` call `try_demolish` directly and stay green regardless of the router precedence. See [camping-crafting-survival.md](camping-crafting-survival.md) § Camp menu routing.
- Demolition refunds the FULL `cost_for(id, biome)` to the bag through the session item API and traces `structure_demolished` (structure_id, tile, refund) plus `materials_refunded` (the symmetric `materials_consumed`), then mirrors `structure_placed` with a `structure_removed` signal so the world view re-renders the tile in place.
- Demolition removes the PLACEMENT entry only (`world_generator.remove_placement`): the tile's clear (if any) lives in the separate clears map and PERSISTS by the two-map v3 design, so demolished ground reverts to cleared open ground — never respawning its tree/rock, never destroying a clear fact.

### The would-trap guard

- `try_place` refuses any solid placement that would leave the player with no step at all: it flood-fills from the PLAYER tile via `world_generator.reachable_walkable_count` with the candidate tile passed as the `blocked` parameter (one extra tile treated as unwalkable — a hypothetical stamp that NEVER mutates the live placements map, so the hypothetical is structural rather than an emergent property of side-effect-free add/remove; budget 2 — self plus one neighbor proves a step exists; 1 means walled in), and refuses with `structure_refused{reason: would_trap}`, consuming nothing. This closes the review-found softlock where a standing player faces N/E/S/W and places a wall on each side (two walls suffice in a 1-wide corridor), ending fully enclosed on their own tile.
- The guard protects PLAYER-tile reachability ONLY: pens intentionally enclose POKEMON, so enclosing a region the player stands outside of is allowed, as is sealing a house whose interior the player can still move in. Walkable structures (the door) never trip the guard. Demolition is the complementary escape: even an enclosed player can Cut their way out with a full refund.
- LOAD-BEARING INVARIANT (region-scale enclosure): the guard does NOT prevent sealing the player inside a BOUNDED REGION — a closed wall ring built around interior space, or a wall dropped on the single land bridge to the island the player stands on, leaves the player tile locally reachable and is ALLOWED by design. Today that is not a permanent softlock only because escape rests on demolition, and demolition capability is guaranteed by a material->move witness chain: every shell class carries a WITNESS material produced ONLY by its demolish move (`log` -> Cut for the default shells, `hard_stone` -> Smash for the desert/sand stone shells), mons never leave the party (no release/box mechanic), and materials come ONLY from harvest (no shop/gift/starting-bag source) — so any party that gathered the materials to build ANY structure can also demolish it. `build_runtime.unwitnessed_demolish_moves()` computes the moves whose shell class has lost their witness (empty today; `placement_flow` asserts it on every run via its demolition half). DO NOT ship a non-harvest material source or a party-release/box mechanic without first extending the guard (e.g. refuse placements that disconnect the player tile from the spawn/outside region, or assert demolition capability at build time) — doing so turns a permitted region seal into a permanent self-trap.

## Persistence

DECISION: placements persist in SAVE SCHEMA V3 via a NEW ADDITIVE top-level key `structures`; the save version STAYS 3. The existing `world_overrides` key (clears) is byte-identical and untouched.

- Two canonical maps, two save keys: the generator's clears map (`world_overrides`, shape `"x,y" -> {kind: "cleared"|"dug", ...}`, unchanged) and a NEW placements map (`structures`, shape `"x,y" -> {kind: "placed", structure_id, by: "build", step}`). They are STORED separately because a tile can carry both facts at once — a structure is placed on a CLEARED tile, and demolition (Cut refunds materials) removes the placement entry so the tile reverts to cleared ground, not its respawned tree. One `"x,y"`-keyed dict cannot hold both.
- Shared 10k cap: `world_overrides` `put` and the new `put_placement` check `_overrides.size() + _placements.size() < 10000`, honoring the existing cap policy and keeping the merged view mirror bounded so the world view's merge never strands an un-rendered tile.
- The world view stays UNTOUCHED: `game_runtime.mutations_for_view()` returns a MERGED mutations view (clears + placements, placements shadowing same-tile clears) via `world_generator.mutations_for_view()`, and `WorldOverrides.apply` branches on entry kind (`cleared`/`dug` → clearing; `placed` → structure). Placing emits the existing `world_overridden(tile)` signal, so the world view re-renders the tile in place — identical to the harvest flow. The SAVE stays split (clears under `world_overrides`, placements under `structures`); the merged accessor is only the view mirror, never the save. Clears-only readers (the `world_consistency_audit` clears lane) use the generator's clears-only `overrides_for_save()` so build placements can never contaminate them.
- Migration is additive, matching the v3 pattern (v3 already grew by ADDING campsite keys without a bump): v1/v2→v3 and pre-building v3 saves backfill an absent `structures` key to `{}`; a building-v3 load validates each entry (`structures` id valid, `"x,y"` key well-formed) and drops invalid entries with a `warning` trace. Phase 3 (future) adds `contents` to box entries — THAT is the v3→v4 bump, which only works because `structures` already exists in v3.

## Scope boundary (documented deviations)

- Bed BEHAVIOR LANDED with Phase 2 (full heal + status cure + revive via interact-rest) and the bed is now BUILDABLE — its `soft_bedding` ingredient is a campfire craft (3 soft_feather + 3 silky_thread, `recipes.gd`), supplied by the interim battle-end material drops. The campfire placeholder is likewise filled (light source + crafting station + runtime `lit` toggle). STORAGE BOX remains a PLACEHOLDER: definition + placement + occupancy + cost only — no contents, no interact; Phase 3 fills it (logged in [../tech-debt-tracker.md](../tech-debt-tracker.md)).
- The "indoors" enclosure flood-fill is STILL DEFERRED: Phase 2 needs indoor gating only for the kiln (itself deferred — no kiln structure exists in any ported phase), so no enclosure computation landed; bed/bag/box are place-anywhere and the sleeping-bag indoor heal bonus is deferred with the flood-fill. See [camping-crafting-survival.md](camping-crafting-survival.md) § Phasing decisions.
- Demolition LANDED with Phase 1 (a review-found permanent-softlock fix, alongside the would-trap guard): Cut refunds all materials, hard-stone shells need Smash. Terrain demolition (Dig on dug ground) stays N/A — no Phase-1 structure modifies terrain; the rule is documented in Demolition above for when it does.
- Per-biome sprite coverage is partial: default + Desert appearance ship now; snow/savanna/spooky variant completeness is tech debt.
- LOAD-BEARING INVARIANT EXTENSION (Phase 2): the demolition-witness chain below now coexists with a SECOND material source — the interim battle-end type drops (`material_drops.gd`: magnet/hard_shell/metal_coat/silky_thread/soft_feather) — and the starting bag now carries a sleeping_bag (not a shell material). The witness HOLDS because the drop table NEVER contains `log` or `hard_stone` (the log→Cut, hard_stone→Smash witnesses stay harvest-only) and the sleeping bag is not a demolition witness; `placement_flow` asserts the invariant against the drop table on every run. The Phase-5 habitat drops must preserve the same exclusion.

## Smoke validation

- `placement_flow` proves refusal without materials (`missing_materials`) and without a capable mon (`not_capable`), then places a wall + a door with exact `structure_placed` + `materials_consumed` payloads (the bag drops exactly the `cost_for` amounts); proves occupancy blocks pathing (the wall tile is non-walkable with a block reason and a step into it is rejected; the door tile is walkable and a step succeeds); and proves persistence across a save round-trip (the reloaded tiles carry the structure sprites and the `structures` map holds both). It earns its logs through the real harvest resolver (built from harvested materials), not a bag grant. Its demolition half (`scripts/app/placement_flow_demolition.gd`, extracted for the app-layer line budget) attempts the EXACT softlock — a standing player walls in all four faced sides — and asserts the first three placements are accepted while the enclosing fourth is REFUSED with `would_trap` (trace payload, no stamped placement, no consumed materials); then it round-trips build->demolish->refund (the wall via the Z-action harvest path, the door via the build runtime): exact `cost_for` refund proven against the bag, `structure_demolished` + `materials_refunded` traces, the tile walkable open ground again, and the wall tile's clear fact intact (read through the build runtime's public `tile_has_clear` seam — demolition never destroys a clear). A Cut-only mon on a hard-stone shell is refused (Smash required) when the pair lands in a desert-shell biome. A dedicated desert round trip (`_check_desert_smash_round_trip`) additionally builds a hard-stone wall on a DESERT/SAND tile and Smash-demolishes it with the exact `hard_stone` charge + refund, so the Smash path is exercised regardless of where the wall/door pair lands (skipped, never failed, when the seed has no desert within reach). Finally it asserts the load-bearing witness invariant (`unwitnessed_demolish_moves` is empty: every shell class has a cost material only its demolish move yields), mechanizing the region-seal escape note above.
- `world_consistency_audit` extends its three-way check to placed structures: each placement is cross-checked across LOGIC (prop sprite + occupancy + block reason), RENDER (tile prop texture present and matching the cache), and COLLISION (a movement probe — wall blocks, door passes). The clears lane is unaffected because it iterates the generator's clears-only save map.
- `save_migration` extends its round-trip with a v3 payload carrying one wall + one door under `structures` (loads and re-saves both) plus a v3 payload WITHOUT `structures` (backfills to `{}`). The v3→v4 fixture lands with Phase 3.
- `visual_sweep` gains a deterministic `13_built_house.png` baseline: under the crafted seed it swaps in a Build-capable mon, crafts the materials, scans outward for the first placeable footprint, and places a fixed wall-ring + door + roof pattern (a small house with a door). Placement is a pure override stamp with no RNG, so the shot is byte-stable.
- DETERMINISM vs LOADED SAVE: because placements now persist under the `structures` save key, the sweep's `craft_state` WIPES the live generator's clears and placements (after writing the seed-only payload, before `rebuild`) so the crafted world is a pure function of the seed. Without this, a house/structures left in the player's save by prior play would be loaded at boot and — through `rebuild` and each `teleport_player`'s rebuild re-pulling `mutations_for_view()` — render into the overworld/biome shots (01-12), drifting them against the baselines that predate the `structures` key. Real play keeps `rebuild` carrying session mutations (`main.gd` rebuilds the overworld on return), so the wipe is sweep-local, not a change to `rebuild`'s contract.
