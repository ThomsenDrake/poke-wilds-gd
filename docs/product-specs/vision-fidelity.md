# Vision Fidelity

This subsystem is the capture-honesty contract for windowed screenshot verification. It never changes what the game renders; it only makes captures trustworthy and their failures legible. Threshold constants and event payloads are single-sourced in `scripts/app/snapshot_capture.gd` (a tree-free `RefCounted`, layer `app`).

## Capture contract

- Every windowed readback first awaits the existing scenario settle waits, THEN awaits a `RenderingServer.frame_post_draw` readback guard (`guard_readback()`), which fires after all viewports finished updating for the frame. The guard is always ADDED AFTER the settles, never a substitute: `visual_sweep.gd:57` `_settle(5)` lets a resized window present before the first capture, and `ui_render_audit.gd:48` warns that the battle SubViewport only redraws while visible, which its `_settle(2)` services. The guard advances whole rendered frames only — no wall-clock dependence — preserving the determinism culture.
- `capture(runtime, viewport, shot, options)` is the shared pipeline: guard → `trace_cursor` (the `SmokeScenarioRunner.trace_log_line_count()` sampled at capture time, before the readback) → `viewport.get_texture().get_image()` → optional `save_png` → byte size → validity oracle → on invalid a `capture_invalid` trace (and an increment of the instance's `invalid` counter), on valid the optional sidecar write (§ Semantic sidecars) plus a `snapshot_captured` trace (now carrying `sidecar_path`) plus the duplicate-capture hook. The sidecar write is file I/O, not a trace: nothing emits a trace between the cursor sample and `snapshot_captured`, so the record still lands at cursor+1. It returns `{ok, image, kind, classification, luminance, bytes, shot_seq, trace_cursor, detail, metadata, sidecar_path}`; the caller owns the returned image and `snapshot_capture` retains nothing.
- `options` accepts `save_path` (default `""` = no PNG write), `shot_seq` (default: the instance's 1-based counter), `dup_check` (scenario-option trigger for the duplicate hook), and `metadata` — the sidecar hook (Slice 3): when non-empty and the capture is valid, `capture()` injects the capture stamps (`shot`, `shot_seq`, `ts_msec`, `trace_cursor`, `window`, `validity`), computes `palettes` from the readback image, and writes canonical `<save_path>.sidecar.json` (schema + correlation protocol: [snapshot-sidecar.md](../references/snapshot-sidecar.md)). `ui_render_audit` and `display_matrix` pass no metadata and write no sidecars.
- Delegation: `visual_sweep.gd` `_capture` delegates and keeps its red-on-invalid windowed contract (an invalid capture still fails the sweep, now counted in `invalid_captures`); it builds `metadata` via `RenderIntrospection.collect(_ctx, filename, crafted)` — the in-process semantic collector consuming the existing `smoke_context()` ctx dict, with NO `main.gd` edits — and `visual_sweep_baselines.gd` co-copies/prunes sidecars with the baseline PNGs. `ui_render_audit.gd` `_pixel_half` delegates the guard plus oracle and (Slice 4) the glyph template match to `scripts/app/text_oracle.gd`, gating every pixel finding per state through `GRADUATED_STATES` (red for graduated battle states, quarantine otherwise; at Slice 4 landing all four keys are false — § Legibility oracles); `display_matrix.gd` `_check_size` inserts the guard between `await _settle(SETTLE_FRAMES)` and the root-viewport `get_image()`. `visual_sweep_passed` gains `window`, `dup_checked`, `sidecar_paths`, and `invalid_captures` (threaded through `visual_sweep_baselines.gd` `report()`).
- Root-viewport-crop fallback: if the battle SubViewport readback is magenta/stale (Godot 4.6 regression #115402 — magenta SubViewport frames after load/save, fixed only in 4.7), `ui_render_audit` crops the root viewport via `crop_battle_display()` (same texel-scale geometry as `display_matrix.gd` `_display_crop`), traces the fallback in the `capture_invalid` detail, and downscales NEAREST to the 160x144 stage so the lint's display contract holds.
- A magenta `capture_invalid` with `classification=regression` firing on the pinned Godot 4.6.1 binary is the guard working, not a defect of this subsystem. The global `tools/visual_diff.py` gate (0 pass / 1 drift / 2 error, 0.5% at tolerance 8) is unchanged by this contract.

## Validity oracle and thresholds

`classify(image, png_bytes)` is pure (no traces). Precedence, first match wins: `headless` > `blank` > `undersize` > `magenta` > `uniform`. Magenta is checked before uniform because a magenta frame is also uniform and magenta is the identified cause.

| Constant | Value | Meaning |
| --- | --- | --- |
| `MIN_SHOT_BYTES` | 5120 | PNG file bytes; below => kind `undersize` (checked only when a PNG was written, i.e. `png_bytes >= 0`) |
| `LUMINANCE_FLOOR` | 0.01 | Rec.709 mean luminance (0-1) over sampled pixels; below => kind `blank` |
| `UNIFORM_LUMA_SPAN` | 0.004 | ~1/255; (max-min) sampled luminance span below => kind `uniform` |
| `MAGENTA_CHANNEL_TOL` | 8 | Sample matches when r >= 255-tol AND g <= tol AND b >= 255-tol (pure #FF00FF signature of #115402) |
| `MAGENTA_RATIO` | 0.5 | Matching sample share at or above => kind `magenta` |
| `SAMPLE_BUDGET` | 4096 | Even-stride pixel samples per validity pass (keeps GDScript cost in low ms at 1152x648) |

Kinds: `headless` = empty image while `DisplayServer.get_name() == "headless"`; `blank` = null/empty image in a windowed run, or mean luminance below the floor; `undersize` = PNG bytes in `[0, MIN_SHOT_BYTES)`; `magenta` = magenta sample share at or above `MAGENTA_RATIO`; `uniform` = luminance span below `UNIFORM_LUMA_SPAN`. Classification is `transport` when the display server is headless and `regression` otherwise; `magenta` is ALWAYS `regression`. Kind `undersize` is also emitted by `capture()` (outside the pure `classify()` above) when `save_png` itself fails on an otherwise-valid image: no PNG bytes exist, `png_bytes` stays -1, and `detail` carries the error code, so a windowed shot that cannot be written still fails red in `visual_sweep`. `capture_invalid` is warning tier: it explains a capture but never fails a scenario on its own (findings policy).

## Transport-honesty semantics

- Headless capture is impossible by engine design (RenderingServerDummy returns blank/dummy values; offscreen rendering is still open proposal #5790). Blank captures under `--headless` are engine reality, classified `transport`, never a red.
- The windowed-only skip set `WINDOWED_ONLY_SCENARIOS = {visual_sweep, visual_sweep_update}` is single-sourced in `tools/godot_dap_smoketest.py` and shared by `tools/run_playtests.py`. `display_matrix` is deliberately NOT in it: it runs under force-headless and self-skips its pixel work in-engine (`display_matrix.gd:44-47`), still emitting `display_matrix_passed {skipped: headless}` — which is also why the default run's harness-skip count is 1, not 2. (`WINDOWED_SUBPROCESS_SCENARIOS`, which does include `display_matrix`, is only the windowed-subprocess LAUNCH set used when NOT force-headless.)
- Under `PLAYTEST_FORCE_HEADLESS=1`, both harnesses report a windowed-only scenario as skipped-with-reason, never failed: the runner prints a SKIP row, the summary line reads e.g. `summary: 19/19 (1 skipped-headless)`, the summary carries `skipped_headless`, and the exit code is 0 whenever only transport skips occurred. In the default 19-scenario run only `visual_sweep` is skipped (`display_matrix` additionally self-skips in-engine with `display_matrix_passed {skipped: headless}` — a pre-existing, distinct skip; `visual_sweep_update` is not in the default set).
- `godot_dap_smoketest.py` run directly with a windowed-only scenario under `PLAYTEST_FORCE_HEADLESS=1` prints `SKIP: <scenario>: windowed-only scenario skipped under PLAYTEST_FORCE_HEADLESS (captures need a real window and renderer)`, writes an `ok: true` result file with `skipped_reason`, and exits 0.

## Duplicate-capture hook

- Triggers: environment `PLAYTEST_CAPTURE_DUPCHECK` set to anything other than `""`/`0`/`false`/`no`/`off`, OR `options.dup_check == true` (the scenario-option path). Default OFF.
- A duplicate check captures the same quiesced shot twice: a SECOND `guard_readback()` plus readback one newly rendered frame later — no game state advances between the pair (the scenario is quiesced), but the engine has rendered one more frame, and frame-to-frame readback differences in that quiesced state are exactly the nondeterminism this hook catches. Both images are converted to `Image.FORMAT_RGBA8` and their `get_data()` byte arrays compared.
- Any nonzero delta emits `capture_invalid` with `kind=nondeterministic_pair`, `classification=regression`, `luminance` from the primary, and a detail of the form `N of M pixels differ; first byte offset K` — the identified cause.
- Quarantine-tier: the primary capture still returns `ok=true`, `dup_checked` increments and feeds `visual_sweep_passed`, the primary PNG stays canonical, and the pair is compared in memory only (never saved).

## Stamp schema

`snapshot_captured` payload (Source `App.SnapshotCapture`, info tier; headless never emits it):

| Field | Type | Meaning |
| --- | --- | --- |
| `shot` | String | Filename (`09_battle.png`) or caller shot name (`battle_action`) |
| `shot_seq` | int | 1-based per SnapshotCapture instance (1..16 across a sweep) |
| `ts_msec` | int | `Time.get_ticks_msec()` at capture time — same boot clock as the record-level `ts_msec` |
| `trace_cursor` | int | Trace-log line count sampled at capture time; join key into `user://logs/agent_trace.jsonl` (this record lands at cursor+1; join with the inclusive `trace_log_has_since(event, cursor, ...)`) |
| `window` | [int, int] | `DisplayServer.window_get_size()` |
| `renderer` | String | `rendering/renderer/rendering_method` project setting (`RenderingServer.get_current_renderer()` does not exist on Godot 4.6.1 — verified; the setting resolves per platform and names the active method, e.g. `forward_plus`) |
| `godot_version` | String | `Engine.get_version_info()["string"]` (the pinned binary reports `4.6.1-stable (official)` — verified from harvested stamps) |
| `sidecar_path` | String | Canonical sidecar written next to the PNG (`res://.../<shot>.png.sidecar.json`, e.g. `res://.godot-smoke/shots/09_battle.png.sidecar.json`); `""` when the capture wrote none (`ui_render_audit`/`display_matrix` pass no metadata). The sidecar mirrors this payload's `ts_msec`/`trace_cursor` — join protocol in [snapshot-sidecar.md](../references/snapshot-sidecar.md) |

`visual_sweep_passed` payload gains (Slice 3, via `visual_sweep_baselines.gd` `report()`): `sidecar_paths` (Array[String] — the `res://.../<shot>.png.sidecar.json` path for each shot) and `invalid_captures` (int — the SnapshotCapture `invalid` counter; previously an invalid capture failed the sweep via `push_error` with no count in the payload. Structurally 0 whenever the trace fires — every invalid capture appends a `_failures` entry that stops the sweep before `report()` runs — so it doubles as a pass-time invariant; the `capture_invalid` trace carries the per-capture count). The per-region verdict fields (`region_failures`, `clusters_explained`, `clusters_unexplained`, `region_artifacts`) are NOT emitted in the trace: the engine never runs the region diff — the playtest runner records them into `.godot-smoke/playtest-report.json` (§ Explainable per-region diff, invocation-point decision).

`capture_invalid` payload (Source `App.SnapshotCapture`, warning tier): `shot` (String), `kind` (`blank`|`uniform`|`magenta`|`undersize`|`headless`|`nondeterministic_pair`), `classification` (`transport`|`regression` — magenta is always regression), `luminance` (float 0.0-1.0, 0.0 when the image is unavailable), `detail` (identified cause: byte count, magenta ratio, luma span, diff-pixel count + first offset, or fallback note).

Report stamps: `.godot-smoke/playtest-report.json` top level carries `head_sha` (`git rev-parse HEAD`), `godot_version`, `window`, and `renderer`, plus `summary` (including `skipped_headless`) and `scenarios`. The three runtime fields are harvested from the last `snapshot_captured` payload seen; absent fields are `null`, NEVER faked (headless-only runs have no `snapshot_captured` traces, so all three are null while `head_sha` is still present). Each run also deletes `.godot-smoke/result-*.json` files older than the run start so stale reds cannot contradict the report. `head_sha` is the freshness-refusal hook for `verify_all.py` (Slice 3+); `check_repo_contracts.py` verifies that, if the report exists, it carries `head_sha` and `godot_version` keys (nulls allowed). The runner also records the Slice 3 region verdict fields (`region_failures`, `clusters_explained`, `clusters_unexplained`, `region_artifacts`) on the `visual_sweep` scenario entry and files non-red drift plus unexplained clusters in a report quarantine section as kind `region_drift` — report-tier, never JSONL.

## Semantic sidecars

Every valid `visual_sweep` capture writes a canonical JSON sidecar `<shot>.png.sidecar.json` next to its PNG — the full field schema, canonical-form policy, and PNG↔sidecar↔`agent_trace.jsonl`↔`playtest-report.json` correlation protocol live in [snapshot-sidecar.md](../references/snapshot-sidecar.md). Sidecars carry the semantic render state an agent needs to explain a pixel diff with no vision: `labels` (visible labels + ink rects in stage and display px), `draw_order` (composed from `world_draw_order.gd`), `palettes` (per-region used colors), `cursor_pairs` (the known-dynamic cursor cells), `expected_regions` (the render model's ink/forbidden/string rects), `canary_rect`, `crafted_state`, `capture_env`, and `validity`.

- Canonical form: `JSON.stringify` (compact, keys sorted recursively — Godot's default, verified on 4.6.1), integer-truncated rects via `UiRenderModel.map_region`, `#rrggbb` palettes sorted lexicographically, no trailing newline. Byte-stable per seed with TWO documented run-varying fields: `ts_msec` (boot clock) drifts across runs (measured: shot 01 ts 1736 vs 1974), and `trace_cursor` (absolute line number into the append-only `agent_trace.jsonl`) shifts each run by exactly the lines the prior run appended (measured: constant delta 43 across all 16 shots between consecutive sweeps — structurally run-varying, not nondeterminism). Byte-stability verification normalizes exactly those two fields and asserts every other byte matches; each must equal the `snapshot_captured` payload's value or the trace join lies (policy single-sourced in [snapshot-sidecar.md](../references/snapshot-sidecar.md) § Canonical form). Committed baseline sidecars are written once per `visual_sweep_update` so git diffs stay clean.
- Injection split: `RenderIntrospection.collect(ctx, shot_name, crafted_state)` builds the semantic fields plus an intermediate `palette_regions` spec; `capture()` injects `shot`/`shot_seq`/`ts_msec`/`trace_cursor`/`window`/`validity` and converts `palette_regions` → `palettes` by scanning the readback image. The collector consumes the existing `smoke_context()` ctx dict — `main.gd` stays untouched.
- Label roots are selected PER SHOT KIND — `UiRenderModel.shown()` stops walking at a Viewport, so a hidden `BattleView` root does not mask labels inside its SubViewport: battle shots (09-12) collect from `BattleViewport/BattleStage` only, menu shots (06-08) from `start_menu` only, overworld shots (01-05) from nothing. Walking `battle_view` outside battle would leak stale battle HUD labels into other sidecars and poison the label join (spurious `label_moved`/`label_deleted`).
- Godot 4.6.1 API caveats (verified on the pinned binary): `Image.get_used_colors()` does not exist ⇒ palettes use a manual distinct-RGB scan (`RenderIntrospection.palette_colors`, capped to canary + hud rects — never full-frame without strided sampling); `RenderingServer.get_video_adapter_driver_info()` does not exist ⇒ `capture_env.driver_info` pinned `[]`; `get_current_renderer()` does not exist ⇒ `capture_env.renderer` reads the project setting; `get_video_adapter_name()` returns `""` under headless (harmless — sidecars are only written on valid windowed captures).
- Baseline sidecars co-copy and prune with the baseline PNGs in `visual_sweep_baselines.gd` (`_update_baselines` copies, the prune pass removes stale `*.sidecar.json`, `clear_shots` removes them), so PNG↔sidecar never desyncs. `check_repo_contracts.py` enforces region coverage: every committed baseline PNG has a sidecar sibling carrying `expected_regions` + `canary_rect` keys, and battle shots 09-12 carry a non-empty `canary_rect` and non-empty `labels`.

## Explainable per-region diff

`tools/visual_region_diff.py` (stdlib-only) joins committed baseline sidecars against fresh shot sidecars for per-region verdicts; `tools/visual_explain.py` (imported by the diff via the same importlib pattern `run_playtests.py` already uses) joins each change cluster to the sidecar for an explanation sentence. Artifacts land in `.godot-smoke/region-diff/`: `clusters.json` (machine-readable cluster bboxes + explanations), `<shot>.ascii.txt` (a 32x18 change-density grid — 1152/36 × 648/36 — an agent reads the heatmap with NO vision), and `<shot>.triptych.png` (baseline | shot | diff via the stdlib encoder `tools/png_canvas.py`). The core tools are stdlib-only — `check_repo_contracts.py` guards this; the optional vision extra stays separate.

Verdict tiers:

- RED — canary: `canary_rect` at tolerance ~0 (strict channel equality) is the FIRST RED-TIER CODED REGION. The gate compares pixels inside the rect, not the rect: a seeded 1-frame strip offset inside the DECIDUEYE canary changes pixels while the rect stays put, and fires red even when the frame delta stays under the 0.5% global gate — exactly the sub-region sensitivity the global gate is blind to.
- RED — ink/string/labels: `expected_regions` ink + string rects and baseline `labels[].display_rect` at near-zero ⇒ `region_ink_lost` / `label_deleted` / `string_drift`. Arming is evidence-based POLICY, not a code flag: these tiers are red only because the duplicate-capture noise floor measured zero across 3 consecutive seeded windowed runs (`PLAYTEST_CAPTURE_DUPCHECK=1`, zero `nondeterministic_pair` traces — evidence on file from the Slice 3 validation pass); before such evidence exists (e.g. on a new machine/driver), the tiers must stay quarantine-tier and the nondeterminism investigated.
- MASKED — known-dynamic zones: the `cursor_pairs` model-cursor cell + live-cursor rects only. The pair's `row` rect is deliberately NOT masked: it spans the whole menu row and covers label ink, so masking it blinds the string-region gate to a deleted/changed move label (this over-mask is exactly what let the seeded deleted-label perturbation slip through until caught in integration). Rows are deterministic — the crafted state pins the cursor selection — so cursor+live is the flake-safe mask. The player back-sprite lunge on `11_battle_after_attack` has no sidecar field today; if the noise-floor or flake pass shows residue there, extend the sidecar with a `dynamic_zones` field and regenerate (cheap — sidecars are git-diffable) rather than hardcoding a mask in the tool.
- BACKSTOP — global gate: the 0.5% / tolerance-8 whole-frame gate is recomputed with semantics identical to `tools/visual_diff.py` (per-channel delta strictly > tolerance) and must agree with the in-engine number. `visual_diff.py` itself — exit-code contract (0 pass / 1 drift / 2 error), global gate, and the in-engine invocation from `visual_sweep_baselines.gd` — is UNTOUCHED, so CI posture is unchanged.
- QUARANTINE — everything else: clusters outside all coded regions below the global gate become kind `region_drift` (quarantine-tier, recorded in the report quarantine section) and enter the explicit unexplained queue.

Explanation vocabulary (`visual_explain.py`): `label_moved`, `label_deleted`, `label_overlap_sprite`, `canary_absent`, `canary_shifted`, `palette_dropped`, `region_ink_lost`, plus the explicit `unexplained` tag. Every cluster gets exactly one explanation or the explicit `unexplained` tag — the unexplained queue is structural and guards false closure (zero unexplained clusters means every change is accounted for).

Invocation point: the region gate is a `tools/run_playtests.py` POST-STEP after the `visual_sweep` scenario (the in-engine `visual_diff.py` global gate runs exactly as today). The runner skips the post-step on `skipped-headless` transport and in `update`/`auto_update` mode (baselines just rewritten ⇒ trivially green); otherwise it calls `run_region_diff(shots_dir=.godot-smoke/shots, baseline_dir=docs/generated/visual-baselines, out_dir=.godot-smoke/region-diff, threshold_pct=0.5, tolerance=8)` and merges `region_failures`/`clusters_explained`/`clusters_unexplained`/`region_artifacts` into the scenario result BEFORE the row prints — non-empty `region_failures` ⇒ the row FAILs ⇒ exit 1. Standalone CLI mirrors `visual_diff.py`: `python3 tools/visual_region_diff.py --shots-dir .godot-smoke/shots --baseline-dir docs/generated/visual-baselines`, exit 0 pass / 1 drift / 2 error. The DAP transport mirrors the post-step in `godot_dap_smoketest.py` (`apply_region_step`, same verdict fields, red-tier failures flip `ok`); DAP is currently down, so the mirror is exercised only when the path is revived.

When to regenerate sidecars: the same triggers as baselines — any `project.godot` rendering/display pin change, a Godot binary bump, a driver/renderer change, or an intentional visual change — because the sidecar is the baseline's semantic twin. Regeneration rides `python3 tools/run_playtests.py --scenario visual_sweep_update`, which co-copies sidecars to `docs/generated/visual-baselines/` and prunes stale ones in the same change; PNG↔sidecar never desyncs. Region gates re-arm only after the 3-run DUPCHECK noise floor is re-measured zero.

## Legibility oracles (Slice 4)

Slice 4 adds three coded legibility oracles: a glyph template match (in-engine, `scripts/app/text_oracle.gd`), a WCAG rendered-pixel contrast check (`tools/contrast_check.py`, stdlib), and a color-vision-deficiency simulation (`tools/cvd_sim.py`, stdlib). The glyph oracle and the contrast check produce coded evidence that can graduate to red; the CVD simulation is accessibility evidence and stays quarantine-tier forever. Findings policy: a glyph mismatch is RED only for states whose `GRADUATED_STATES` key is true, else quarantine; `contrast_low` graduates to coded only after its determinism proof plus a clean-run history; `cvd_collapse` is never red. All three are deterministic: the glyph oracle matches against a noise floor measured on the pinned binary (table below), the contrast check is a pure function of pixels, and the CVD matrices are hardcoded cited constants.

### Glyph template oracle (`scripts/app/text_oracle.gd`)

A tree-free `RefCounted` (layer `app`, preloads `ui_render_model.gd` — same-layer, so the dependency rule stays green) that rasterizes every `UiRenderModel.expected()` string into an ink mask via TextServer glyph-atlas readback and XOR-matches it at the modeled rect against the engine readback. A failed match IS the garble signal — no heuristic, and per-glyph bboxes + min-stroke-width checks come out of the same readback.

- Raster pipeline (the shape verified working on the pinned Godot 4.6.1 binary): a `Node2D` whose `_draw()` calls `Font.draw_string(get_canvas_item(), pen, text, HORIZONTAL_ALIGNMENT_LEFT, -1.0, 7, Color.BLACK)` per string (same `res://pokewilds/fonts.ttf` @7 the battle Labels use, via `UiRenderModel.FONT_PATH`/`FONT_SIZE`) is parented into a throwaway `SubViewport` (e.g. 160x32, `transparent_bg=false`, `render_target_update_mode=UPDATE_ALWAYS`, white `ColorRect` background) added as a child of the host audit node. After `await host.get_tree().process_frame` + `await RenderingServer.frame_post_draw`, the mask is read back with `RenderingServer.texture_2d_get(subvp.get_texture().get_rid())` (converted to `FORMAT_RGBA8`; `get_image()` as the empty-fallback — the two readback routes agree byte-for-byte on the mask, measured 0 mismatch), and the SubViewport is freed. WHY `Node2D._draw` and not a bare `RenderingServer.canvas_item_create()` + `canvas_item_set_parent(ci, vp.find_world_2d().canvas)`: the bare server-created canvas item read back BLANK (0 ink) in the grounding experiment — `Font.draw_string` only commits its textured glyph quads inside a `_draw` context, so the Node2D/`_draw` shape is the canonical one (`SubViewport.get_canvas()` does NOT exist in 4.6.1).
- Binarization shares the visual_lint ink predicate: `ink = (r*299+g*587+b*114)//1000 < 128` — the identical ink definition `tools/visual_lint.py` uses, so the oracle's garble/low_ink vocabulary agrees with the lint's by construction.
- Per-glyph pen/bbox: cumulative `Font.get_string_size(text.substr(0, i), HORIZONTAL_ALIGNMENT_LEFT, -1.0, 7).x` gives the exact pen x each glyph rasterized at (kerning included); equivalently `TextServer.shaped_text_get_glyphs` offset/advance (confirmed present on the binary). Per-glyph min-stroke width = the minimum horizontal ink run-length of the readback inside the glyph bbox.
- Raster parity the oracle inherits for free (so its raster equals the battle Label's): `ts.font_get_global_oversampling() = 1.0` (process-global — the throwaway SubViewport and the battle SubViewport share it; the loaded font's own oversampling 0.0 means "use the global", also shared), battle font `subpixel_positioning = AUTO` (runtime enum value 1 — the `.import` value 4 maps to AUTO at runtime; 4.6.1's enum is DISABLED=0, AUTO=1, ONE_HALF=2, ONE_QUARTER=3 — there is NO ONE_EIGHTH constant), AUTO resolving to ONE_QUARTER at 7px, `hinting = LIGHT`, `antialiasing = GRAY`, `Font.get_ascent(7) = 7.0`, `Font.get_height(7) = 9.0`, plus the Slice 2 pins `snap_2d_transforms_to_pixel=true` (quantizes the Label transform to an integer pen) and `msaa_2d=0`. The oracle must NOT call `ts.font_set_oversampling` or set a per-viewport texture-filter override, or its raster diverges from the battle Label's — touch nothing.
- API: (1) `collect(host, state, model) -> Dictionary` (async — parks the throwaway SubViewport on the audit node, draws every expected string at its model pen (integer baselines = model rect position + ascent), reads back and binarizes, and returns `{mask: Image, glyphs: [...]}`, each glyph carrying `text`, `mode`, the string `rect`, per-glyph bboxes (`glyph_rects` from cumulative `get_string_size`), ink count, and `min_run` — the min horizontal ink run-length — with the result cached for the later synchronous calls); (2) `check(state, model, collected, image, display) -> Array` (synchronous — maps each model rect into image space via `UiRenderModel.map_region`, crops engine mask and collected mask per glyph, XORs; matches ANCHOR-mode strings only, so the box-mode `battle_message` string is never glyph-matched — its pixel evidence is the lint; emits `{kind: "glyph_mismatch", state, text, glyph_index, region, mismatch}` when a glyph's XOR exceeds `T_glyph` or a string's total XOR exceeds `T_str` (`glyph_index: -1` marks a string-total finding), plus corroborating `low_ink` (glyph ink present in the collected mask, zero in the engine crop), `garble` (engine max run-length ≥ `SMEAR_RUN = 3` px — 3x the healthy 1px stem — while the collected glyph stays under it), and `clipped` (glyph bbox extends past the 160x144 stage) findings); (3) `pixel_findings(state, model, image, display) -> Array`, the convenience the audit calls: `UiRenderModel.run_lint(state, model, image, display)` (the existing ink/forbidden/garble bridge — its CALL SITE relocates here from `ui_render_audit`, the sanctioned delegation that keeps the audit ≤220 lines) plus `check()` merged into ONE findings list so the audit loops once. `text_oracle` itself emits NOTHING (no runtime handle); the audit owns all trace emission.
- On the magenta `#115402` fallback path the image checked is the root-viewport crop NEAREST-resized to 160x144 — at integer scale 4 that downsamples block-uniform to exact stage pixels, so the mask equals the native 1:1 raster and the same tolerance holds. ### Tolerance calibrated from measured noise

Measurement protocol: TWO separate process launches of the pinned binary (sequential, never concurrent) rasterize the 17 battle strings covering the full `expected()` glyph repertoire (FIGHT PKMN ITEM RUN PECK GROWL DECIDUEYE :L 18 61/61 PSN NORMAL POKE BALL x99 POTION x99 BACK 20 "A wild DECIDUEYE appeared!" TYPE/) at 7px on 12px-pitched bands, so per-band mismatch maps 1:1 to a string; XOR analysis decodes the PNGs via the repo's stdlib PNG decoder (same importlib pattern as `tools/visual_lint.py`). Measured on Godot `4.6.1-stable (official)`, renderer `forward_plus`, Apple M4:

| Comparison | Mismatch (ink px) |
| --- | --- |
| Cross-launch raster drawA vs drawB | 0 (1576 vs 1576) |
| Cross-launch Label labelA vs labelB | 0 (1576 vs 1576) |
| Raw draw_string vs in-engine Label, white bg (the equivalence gap) | 0 (1576 vs 1576) |
| Raw draw_string vs Label on the action-box fill #f8f8f8 | 0 (1582 vs 1582) |
| Cross-launch on #f8f8f8 | 0 |
| White-vs-#f8f8f8 binarization fringe (the ONLY thing that moved pixels) | 6 px TOTAL, max 1 px per string (`:L 18`, NORMAL, POKE BALL x99, POTION x99, `20`, "A wild DECIDUEYE appeared!" flip 1 each; FIGHT/PKMN/ITEM/RUN/PECK/GROWL/DECIDUEYE/PSN/BACK/TYPE/ flip 0) |
| Min horizontal ink run-length (healthy stem width) | 1 px for every string, both rasters |

The DUPLICATE-RUN RASTER-EQUIVALENCE PROOF — the raw TextServer raster matches the in-engine rendered Label at 7px across repeated windowed runs — is therefore LANDED on this machine (0 gap draw==label, 0 cross-launch, on both white and #f8f8f8).

Tolerance rule (calibrated from these numbers, never assumed): the oracle draws expected glyphs black-on-white (simplest — no static-stack reproduction) and compares to the engine readback mask with a per-string budget `T_str = max(2, 2*N_str)` where `N_str` = the measured per-string equivalence noise = max(0 cross-launch, 0 draw-vs-label, 1 white-vs-#f8 fringe) = 1 ⇒ **T_str = 2 px/string**, and a per-glyph budget `T_glyph = max(1, 2*N_glyph)` with `N_glyph = 0` ⇒ **T_glyph = 1 px/glyph**. Rationale: 2 px/string and 1 px/glyph give 2x headroom over the measured worst-case single AA-fringe flip, while a genuine garble/missing/clipped 7px glyph changes many pixels (a single dropped glyph loses ~10-40 ink px; a wrong glyph shifts a whole column) and trips the budget decisively. The budgets are calibrated-WITH-headroom, not exact-zero, exactly as the spec demands.

Re-measurement protocol: after ANY `project.godot` rendering pin, font `.import` parameter, or Godot binary change, re-run the two-launch XOR under a MATCHING `godot_version`/`renderer` stamp; if N rises, raise T from the new N — never hardcode. Refuse to compare a calibration measured under a different stamp: hinting=LIGHT + GRAY AA are deterministic within the pinned binary (measured 0 cross-launch) but driver/binary-specific (MoltenVK/Vulkan here vs `d3d12` on Windows). A sub-pixel risk to watch: the equivalence holds because the model anchors are integer AND `snap_2d_transforms_to_pixel=true` quantizes the Label transform (measured pen fractional part = 0). A future layout placing a battle label at a fractional position would spike the white-bg XOR — treat that as a REAL layout signal, not oracle flake (the per-glyph T=1 / per-string T=2 absorbs a single fringe flip, not a systematic 1/4px shift across a whole string).

Alternative (future tightening — documented, not required): composite expected glyphs on the SAME baked-art + overlay + glyph-cover tint the engine uses per state (load `res://pokewilds/battle/battle_bg1.png` + the state overlay `res://pokewilds/battle/battle_screen2.png` or `res://pokewilds/menu/attack_screen1.png` + a `ColorRect` tinted `Color(0.973, 0.973, 0.973)` for action / `Color(1.0, 0.996, 1.0)` for moves-item, matching `battle_surface._glyph_covers.modulate`) — then the white-vs-#f8 fringe disappears and T can drop to 0; the 0-gap-on-matching-bg measurement above is the evidence that path is exact. A state whose real background is darker than #f8 (a future tinted overlay) could push more AA fringe across the 128 threshold and exceed T_str=2; if measurement shows per-string fringe >2 against white, switch THAT state's oracle to the composite path.

### WCAG rendered-pixel contrast (`tools/contrast_check.py`)

Stdlib-only (the `core_tools_stdlib_issues` guard enforces this; PNG decode imports `tools/visual_diff.decode_png_rgba` via importlib, the same pattern as `visual_lint.py`). A PURE function of pixels: for a label rect, sample fg = the median of the sorted DISTINCT ink colors (lum<128 — the text ink) and bg = the median of the sorted DISTINCT fill colors (lum≥128 — the box fill), which understates true contrast slightly and so keeps the check conservative. Linearize each channel: `cl = c/255; lin = cl/12.92 if cl <= 0.04045 else ((cl+0.055)/1.055)**2.4`; relative luminance `L = 0.2126*Rlin + 0.7152*Glin + 0.0722*Blin`; ratio `(max(Lfg, Lbg) + 0.05) / (min(Lfg, Lbg) + 0.05)`. 7px battle text is WCAG "normal" (not ≥18pt, not ≥14pt bold) ⇒ AA threshold **4.5:1**; the large-text 3:1 is kept as a parameter for any future ≥18pt label. Emits `{kind: "contrast_low", state/shot, label_text, rect, ratio, need: 4.5}` when ratio < 4.5 — catching text-over-battle-effect contrast loss the design-time checks miss.

Determinism (exit criterion 3): the function is a pure reduction over pixels with stable per-rect ordering (sort distinct colors, take the median of a sorted list — no RNG, no float-ordering nondeterminism), so identical input bytes ⇒ identical ratios. The 3-run proof: run it on 3 consecutive windowed captures which Slice 2 proved byte-identical (2/2 bit-identical sweeps, max drift 0.0) ⇒ zero run-to-run variance BY CONSTRUCTION; it also trivially has zero variance when pointed at the committed baseline PNGs (constant input). `contrast_low` is quarantine-tier and graduates to coded RED only AFTER the determinism proof (3 identical runs) plus a clean-run history, exactly parallel to the glyph match. The recorded 09-12 ratios span **5.147–8.404** (1.14x–1.87x headroom); the tightest label is **'35' on 10_battle_moves at 5.147 (1.14x)**, and the median-distinct fg UNDERSTATES true contrast (it is an AA-fringe gray, not the pure-black core — a black/#f8 pair is ~17:1), so the promotion decision must re-measure that low-end headroom (or anchor fg to a low percentile of ink luminance, or hold the thin-margin labels quarantine-tier) alongside the clean-run history the rule already requires.

CLI + invocation: `--image P --rects J --out O` for a single PNG plus a rects JSON (`[{"text": ..., "rect": [x,y,w,h]}]`), or `--shots-dir D --out O` for a directory whose PNGs carry sibling `.sidecar.json` files supplying the label rects; `--large` switches to the 3:1 large-text threshold (mirrors `visual_lint.py`'s shape). Two callers feed it: (i) DESIGN-TIME — the new "Contrast" section of `tools/generate_legibility_report.py` reads the committed `docs/generated/visual-baselines/09_battle.png` (+10/11/12) pixels and the per-label `display_rect` from the sibling `.sidecar.json` `labels[]` (battle shots carry non-empty labels per the region-coverage check) — no Godot needed; (ii) FRESH PIXELS — a `run_playtests.py` post-step on `visual_sweep`, modeled exactly on `apply_region_gate` (lazy importlib load; skipped on transport-skip and in update/auto_update mode), reads the fresh `.godot-smoke/shots/<shot>.png` + fresh sidecar `labels[]` and files findings in the report quarantine section as kind `contrast_low` — report-tier, never JSONL, matching the region-diff invocation-point decision. This fresh path is what catches a new translucent effect drawn over a label (committed baselines cannot show it). For `ui_render_audit` (which writes no sidecar) the post-step can instead consume `.godot-smoke/lint/<state>.png` (160x144) + the model strings mapped at scale 1.

### Color-vision-deficiency simulation (`tools/cvd_sim.py`)

Stdlib-only. Hardcodes the THREE severity-1.0 Machado matrices — the paper's published values with a citation, NOT an unverified gist:

- Citation: G. M. Machado, M. M. Oliveira, L. A. F. Fernandes, "A Physiologically-based Model for Simulation of Color Vision Deficiency," IEEE Transactions on Visualization and Computer Graphics 15(6), 2009, DOI 10.1109/TVCG.2009.113. The verbatim severity-1.0 ladder values derive from G. M. Machado's 2010 MSc thesis (LUME UFRGS repository, handle 10183/26950) and were cross-checked against the colour-science tabulated dataset and colorspacious's [MOF09] matrices. The citation comment in `cvd_sim.py` must carry BOTH references.
- Severity 1.0 (full dichromacy) is hardcoded as the worst-case accessibility bound; the 0.0-1.0 ladder can be added later but is not needed for the quarantine evidence. Each matrix's rows sum to 1.0 — asserted at import as a white-preservation self-check AND a tamper check against a future maintainer swapping in unverified values.

Application to a palette (list of `#rrggbb`): linearize each color, multiply by the 3x3 matrix, clamp [0,1], re-gamma to sRGB. Collapse detection: a pair "collapses" under a deficiency when its original CIE76 deltaE is at or above `DELTA_E_THRESHOLD = 10.0` (the "clearly distinguishable" boundary, calibrated on the HP-bar triple — the green/red pair's original deltaE 109.2 collapses below the threshold under deutan (simulated 8.14; protan 34.29 and tritan 118.7 stay above)) but the simulated deltaE falls below it. Emits `{kind: "cvd_collapse", deficiency: "protan"|"deutan"|"tritan", pair: [a, b], source, original_delta_e, simulated_delta_e, threshold}`. Gray pairs self-filter: a gray (R==G==B) is fixed by the white-preserving matrix, so gray/gray pairs keep their deltaE and can never trip the test — only chromatic pairs can collapse.

Target palettes: the HP-bar triple from `battle_surface._set_hp_bar` — green `Color(0.227, 0.651, 0.247)` = `#3aa63f`, yellow `Color(0.929, 0.733, 0.118)` = `#edba1e`, red `Color(0.816, 0.239, 0.204)` = `#d03d34` (the classic green/red CVD hazard) — the status-color set, and the committed sidecar `palettes.canary` lists (e.g. 09's canary palette — sidecars carry palettes directly, so the sim needs no pixel decode).

Routing: `cvd_collapse` is QUARANTINE-FOREVER — accessibility evidence, recorded in the legibility report and in the report quarantine section by the `visual_sweep` runner post-step (`apply_contrast_cvd`, alongside `contrast_low`). Never red, never self-graduating; no clean-run streak can flip it, and no CVD *finding* can turn the suite red (a cvd_sim tool *error* still fails red, fail-closed like every other gate). Because palettes are byte-stable per seed, the simulation is identical on fresh vs committed sidecars — the report-generator path is sufficient, and the runner post-step's CVD half exists for completeness.

### Default reviewer (deterministic sidecar-consistency)

The DEFAULT REVIEWER is a deterministic in-process sidecar-consistency checker — no model, CI-safe, a pure function of sidecar + clusters bytes → byte-stable findings per seed. Per changed shot it consumes the fresh sidecar (`.godot-smoke/shots/<shot>.png.sidecar.json`), the committed baseline sidecar (`docs/generated/visual-baselines/<shot>.png.sidecar.json`), that shot's clusters entry from `.godot-smoke/region-diff/clusters.json`, the assembled bundle, and the rubric excerpt — receiving the SAME stdin dict a `--reviewer-cmd` plugin would, honoring the interface. Every finding cites a region BY CONSTRUCTION because each is generated FROM a sidecar field with `bbox` = that field's rect(s): `label_deleted` / `label_moved` / `label_text_changed` (labels by array index), `cursor_missing` / `cursor_moved` / `cursor_appeared` (cursor_pairs by id), `draw_order_changed` (only when the draw rect is groundable), `palette_dropped` (canary palette only — hud is ungroundable), `canary_rect_changed`, `expected_region_changed` (ink/string rect-list deltas), and exactly one `cluster_unexplained` per unexplained `clusters.json` cluster (region_id = the most-specific groundable region intersecting the cluster bbox; intersects nothing → counted as `ungroundable_clusters`, never faked). Deterministic severity/confidence map: sidecar deltas medium/high, canary + palette high/high, clusters low/medium. It emits NOTHING for byte-identical shots and NOTHING groundable for overworld/menu shots lacking regions — the coverage gap is counted in the grounding stats, not faked (per-shot-kind coverage: battle 09-12 all kinds, menu 06-08 `label:<i>` only, overworld 01-05 zero groundable regions — see the rubric's region-id namespace).

REVIEWER INVOCATION: a `--reviewer-cmd` plugin gets stdin JSON `{shot, shot_kind, paths{before, after, som_before, som_after, crops[], expected_strings, rubric, context}, reviewer_params, finding_schema, region_table, window, clusters, som_legend, grounding_rules}` and must return stdout `{"findings": [...]}` — subprocess with `shlex.split`, NO `shell=True`, hard timeout (default 300s); non-zero/timeout/invalid JSON = tool error → exit 2 standalone / runner exception red, fail-closed like every other gate (a hung or garbage plugin is never a silent pass). `window` (frame bounds), `clusters` (change regions), and `som_legend` (the SoM number→region_id join) ride in stdin — not only the on-disk `context.json` — so the plugin can build in-frame, groundable bboxes from stdin alone. The deterministic default is an in-process function with the SAME stdin-dict → findings-list signature — the interface is honored, the subprocess is skipped. Returned findings then go through schema validate/repair → grounding enforcement → `finding_id` minting → write. Reviewer discipline is RECORDED in the output (and shipped to plugins in the stdin `reviewer_params`): temperature 0, n=2 unanimity vote (both passes must emit), before/after order shuffle — an honor-system contract for model plugins (the pipeline calls a plugin ONCE per shot and mechanically enforces grounding on the result, not the vote); the deterministic default runs ONCE (votes are meaningless when deterministic) but honors the interface (see the rubric's reviewer-parameter block).

### Grounding enforcement

Pipeline-side, after ANY reviewer returns, before `vision-review.json` is written: a finding is GROUNDED iff it carries a `region_id` resolvable in the shot's region table (the union of the FRESH + BASELINE sidecars, addressed per the rubric's region-id namespace) AND its integer `bbox` `[x,y,w,h]` (native display px, w>0, h>0, inside the window frame) passes `visual_explain.rects_overlap` against at least one rect registered to that id. `rects_overlap` is imported from `tools/visual_explain.py` via the importlib pattern — the same function `visual_region_diff._classify` uses, never re-implemented. `draw:<node>` rects are STAGE px and are first mapped to display px with the documented `battle_view` integer-scale formula (`k = floor(min((w-32)/160, (h-32)/144))`, centered in `window`), runtime-cross-checked (the mapped `draw:Cursor` rect must equal the `cursor_pairs` live rect whenever the cursor is visible — a mapping bug would silently unground every draw finding). DROP-AND-COUNT: findings failing region resolution, intersection, in-frame bbox, or unrepairable schema validation are excluded from the emitted array and counted in file-level grounding stats `{emitted, grounded, dropped, dropped_reasons: {unknown_region_id, no_intersection, schema_invalid, bbox_out_of_frame}, dropped_samples[<=8]}`. 100% of EMITTED findings are grounded BY CONSTRUCTION — enforcement is the gate, not a lint; a mock reviewer's ungrounded garbage is dropped + counted with exit still 0 (drops are not errors). Regions without a rect (overworld draw nodes with rect [], `palette:hud`) are UNgroundable — deltas there are recorded in bundle context and counted as `ungroundable_deltas`, never as findings: silence is counted, never lost.

### Finding schema and finding_id (Slice-6 ledger join)

The output REPLACES the stale pilot's bare array: schema `vision-review/2` — top level `{schema, generated_by, generated_at, head_sha, rubric_ref, reviewer{kind, cmd, params}, manifest{window, shots_covered[]}, grounding{...stats...}, shots[]}`; the first run unconditionally deletes/overwrites the gitignored July-2026 pilot (its array shape fails validation and is replaced, not repaired). Per finding, the rubric fields (`shot`, `class`, `region`, `severity`, `confidence`, `note`) are kept and additively extended with `region_id`, `bbox` (finding extent, must intersect `region`), `explanation` (a sentence citing the sidecar evidence), `evidence_crop` (bundle-relative base|fresh twin path), `sidecar_ref` (`{source: baseline|fresh|both, field, baseline, fresh}` — the auditable join), `reviewer_kind`, and `finding_id`. Repair rules: severity/confidence coerced to enum (bad → medium/low + repair note), bbox int-coerced; missing/unknown region_id or an unrepairable bbox → DROP (counted). UNCHANGED shots still get a `shots` entry (`changed: false`, `findings: []`) so the manifest is the complete freshness authority. Full field-level schema: [references/vision-review-rubric.md](../references/vision-review-rubric.md) § Finding schema.

`finding_id` = `vr1-` + the first 16 hex chars of sha256 over canonical JSON (`json.dumps(sort_keys=True)`, compact separators) of exactly `{v: 1, shot, class, region_id, bbox, sidecar_field, baseline, fresh, cluster}` — DELIBERATELY EXCLUDING timestamps, head_sha, file paths, reviewer kind, severity/confidence, and evidence_crop, so the same defect at the same shot/region/values hashes identically across re-runs, transports, and reviewer implementations (a model reviewer and the deterministic default reporting the same defect JOIN in the ledger). `finding_id` is minted POST-repair by the pipeline — reviewers never mint it. The versioned prefix lets Slice 6 rotate the scheme without silent collisions; baseline regeneration changes baseline/fresh values and hence ids — the ledger treats a baseline regen as a new epoch (the baseline sidecar sha is carried in `sidecar_ref` context for audit, not in the hash). The runner records the quarantine→graduation keys: every report quarantine entry of kind `vision_review` carries `finding_id`, so Slice 6's `graduation_ledger.py` joins a graduated coded oracle's finding to the quarantine finding it confirms.

### Runner post-step (`apply_vision_review`)

`run_playtests.py` runs `apply_vision_review` on the `visual_sweep` scenario entry AFTER `apply_region_gate` (which produces the `clusters.json` the review consumes) and `apply_contrast_cvd`, modeled exactly on those steps: skipped on `skipped-headless` transport and in `update`/`auto_update` mode (baselines just rewritten ⇒ trivially green); exceptions fail red fail-closed; findings NEVER flip `ok`. It REGENERATES `.godot-smoke/vision-review.json` on EVERY visual_sweep compare run — even with 0 changed shots (full manifest, zero findings) — so any sweep whose shots change necessarily has a current file (exit criterion 1). It records `vision_review_written` `{path, shots_reviewed, shots_changed, findings, grounded, dropped, reviewer}` on the scenario entry (`shots_reviewed` = shots COVERED by the review manifest — every baselined shot, changed or not, since the manifest is the freshness authority; `shots_changed` = the subset whose fresh bytes differed from baseline — the shots actually re-reviewed, 0 on a zero-drift sweep) plus kind-`vision_review` quarantine entries (each carrying `finding_id`) in the report quarantine section — report-tier only, NEVER JSONL (the engine never runs the vision review). Failures-section lines print the dropped/ungrounded summary. Runtime self-reference guard: if `apply_region_gate` errored, `clusters.json` may be stale or absent — the review degrades to sidecar-delta-only findings plus a recorded warning, never crashes (quarantine tier).

### Staleness refusal

`manifest.shots_covered[]` carries the sha256 of each FRESH PNG's bytes (content hash — mtime-independent) plus the baseline + both sidecar hashes — the freshness authority. `review_is_fresh(review_doc, shots_dir, baseline_dir)` returns false when any covered shot's current bytes mismatch ANY hash its manifest entry recorded (the fresh PNG sha always; the baseline PNG + both sidecar shas whenever recorded; a recorded hash whose file is gone is stale; hashes the builder could not record are skipped), or a shot in shots_dir with a baseline PNG is absent from the manifest — the shot-scope predicate is the manifest builder's own (has-baseline-PNG), so the two scopes coincide by construction; `baseline_dir=None` keeps the legacy has-fresh-sidecar scope + fresh-PNG-only comparison for callers that know only the shots dir, and the runner post-step passes `baseline_dir` so all four recorded hashes gate freshness there. Because the post-step regenerates on every compare run, staleness cannot persist after a sweep; if the post-step was SKIPPED (a transport-skip run) and a stale file exists on disk, the runner REFUSES it — does not record `vision_review_written`, prints a warn — and `verify_all.py` (L.1, future) refuses-on-mismatch the same way the report `head_sha` hook does. Lane-4 staleness is never red (quarantine tier): refuse = ignore + warn + regenerate, never fail. Only a `vision_review` tool ERROR (bad PNG decode, reviewer subprocess timeout/non-zero/invalid JSON, unwritable output) fails red via the runner exception path (exit 2 standalone).

## Determinism knobs (Slice 2, verify-first)

The game scales MANUALLY in two independent places — `battle_view.gd` `_layout_display()` computes an integer scale `k = floor(min((w-32)/160, (h-32)/144))` for the 160x144 battle SubViewport (Nearest-forced upsample), and the overworld camera zooms 3x over 16px tiles — with NO engine stretch (`display/window/stretch/mode` is `disabled`; `project.godot` carries no `[display]` section). | Project setting key | Pinned value | Engine default (4.6.1, measured via `property_get_revert` on the pinned binary) | Effect |
| --- | --- | --- | --- |
| `rendering/anti_aliasing/quality/msaa_2d` | `0` | `0` (disabled) | Documents the already-disabled 2D MSAA so a future editor toggle cannot silently add per-sample blending to pixel art. Zero rendering delta by construction. |
| `rendering/2d/snap/snap_2d_transforms_to_pixel` | `true` | `false` | Quantizes 2D transforms to screen pixels at sampling time: the overworld walk lerp (`player_avatar.gd` position lerp, no camera smoothing) and any fractional camera position land on integer pixel offsets instead of smearing tile edges through the inherited Linear canvas filter. `rendering/2d/snap/snap_2d_vertices_to_pixel` stays at its default `false` (not pinned). |

`tools/determinism_verify.py pins` asserts both pins are present with these values AND that the rejected candidate pins below stay absent; it is the permanent verify-first aid (absorbed by `verify_all.py` when it lands).

Editor-strip runbook: the Godot editor DROPS settings equal to engine defaults when it rewrites `project.godot` (any editor open/import pass) — `msaa_2d=0` is a default and has been stripped by an editor pass in practice, while the non-default `snap_2d_transforms_to_pixel=true` survives.
