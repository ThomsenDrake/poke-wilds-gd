# Vision Fidelity

This subsystem is the capture-honesty contract for windowed screenshot verification. It never changes what the game renders; it only makes captures trustworthy and their failures legible. Threshold constants and event payloads are single-sourced in `scripts/app/snapshot_capture.gd` (a tree-free `RefCounted`, layer `app`).

## Capture contract

- Every windowed readback first awaits the existing scenario settle waits, THEN awaits a `RenderingServer.frame_post_draw` readback guard (`guard_readback()`), which fires after all viewports finished updating for the frame. The guard is always ADDED AFTER the settles, never a substitute: `visual_sweep.gd:57` `_settle(5)` lets a resized window present before the first capture, and `ui_render_audit.gd:48` warns that the battle SubViewport only redraws while visible, which its `_settle(2)` services. The guard advances whole rendered frames only — no wall-clock dependence — preserving the determinism culture.
- `capture(runtime, viewport, shot, options)` is the shared pipeline: guard → `trace_cursor` (the `SmokeScenarioRunner.trace_log_line_count()` sampled at capture time, before the readback) → `viewport.get_texture().get_image()` → optional `save_png` → byte size → validity oracle → on invalid a `capture_invalid` trace (and an increment of the instance's `invalid` counter), on valid the optional sidecar write (§ Semantic sidecars) plus a `snapshot_captured` trace (now carrying `sidecar_path`) plus the duplicate-capture hook. The sidecar write is file I/O, not a trace: nothing emits a trace between the cursor sample and `snapshot_captured`, so the record still lands at cursor+1. It returns `{ok, image, kind, classification, luminance, bytes, shot_seq, trace_cursor, detail, metadata, sidecar_path}`; the caller owns the returned image and `snapshot_capture` retains nothing.
- `options` accepts `save_path` (default `""` = no PNG write), `shot_seq` (default: the instance's 1-based counter), `dup_check` (scenario-option trigger for the duplicate hook), and `metadata` — the sidecar hook (Slice 3): when non-empty and the capture is valid, `capture()` injects the capture stamps (`shot`, `shot_seq`, `ts_msec`, `trace_cursor`, `window`, `validity`), computes `palettes` from the readback image, and writes canonical `<save_path>.sidecar.json` (schema + correlation protocol: [snapshot-sidecar.md](../references/snapshot-sidecar.md)). `ui_render_audit` and `display_matrix` pass no metadata and write no sidecars.
- Delegation: `visual_sweep.gd` `_capture` delegates and keeps its red-on-invalid windowed contract (an invalid capture still fails the sweep, now counted in `invalid_captures`); it builds `metadata` via `RenderIntrospection.collect(_ctx, filename, crafted)` — the in-process semantic collector consuming the existing `smoke_context()` ctx dict, with NO `main.gd` edits — and `visual_sweep_baselines.gd` co-copies/prunes sidecars with the baseline PNGs. `ui_render_audit.gd` `_pixel_half` delegates the guard plus oracle and keeps `GRADUATED := false` (findings stay quarantine-tier); `display_matrix.gd` `_check_size` inserts the guard between `await _settle(SETTLE_FRAMES)` and the root-viewport `get_image()`. `visual_sweep_passed` gains `window`, `dup_checked`, `sidecar_paths`, and `invalid_captures` (threaded through `visual_sweep_baselines.gd` `report()`).
- Root-viewport-crop fallback: if the battle SubViewport readback is magenta/stale (Godot 4.6 regression #115402 — magenta SubViewport frames after load/save, fixed only in 4.7), `ui_render_audit` crops the root viewport via `crop_battle_display()` (same texel-scale geometry as `display_matrix.gd` `_display_crop`), traces the fallback in the `capture_invalid` detail, and downscales NEAREST to the 160x144 stage so the lint's display contract holds.
- A magenta `capture_invalid` with `classification=regression` firing on the pinned Godot 4.6.1 binary is the guard working, not a defect of this subsystem. The global `tools/visual_diff.py` gate (0 pass / 1 drift / 2 error, 0.5% at tolerance 8) is unchanged by this contract.

## Validity oracle and thresholds

`classify(image, png_bytes)` is pure (no traces). Precedence, first match wins: `headless` > `blank` > `undersize` > `magenta` > `uniform`. Magenta is checked before uniform because a magenta frame is also uniform and magenta is the identified cause.

| Constant | Value | Meaning |
| --- | --- | --- |
| `MIN_SHOT_BYTES` | 5120 | PNG file bytes; below => kind `undersize` (checked only when a PNG was written, i.e. `png_bytes >= 0`) |
| `LUMINANCE_FLOOR` | 0.01 | Rec.709 mean luminance (0-1) over sampled pixels; below => kind `blank` |
| `UNIFORM_LUMA_SPAN` | 0.004 | ~1/255; (max-min) sampled luminance span below => kind `uniform` |
| `MAGENTA_CHANNEL_TOL` | 8 | Sample matches when r >= 255-tol AND g <= tol AND b >= 255-tol (pure #FF00FF signature of #115402) |
| `MAGENTA_RATIO` | 0.5 | Matching sample share at or above => kind `magenta` |
| `SAMPLE_BUDGET` | 4096 | Even-stride pixel samples per validity pass (keeps GDScript cost in low ms at 1152x648) |

Kinds: `headless` = empty image while `DisplayServer.get_name() == "headless"`; `blank` = null/empty image in a windowed run, or mean luminance below the floor; `undersize` = PNG bytes in `[0, MIN_SHOT_BYTES)`; `magenta` = magenta sample share at or above `MAGENTA_RATIO`; `uniform` = luminance span below `UNIFORM_LUMA_SPAN`. Classification is `transport` when the display server is headless and `regression` otherwise; `magenta` is ALWAYS `regression`. Kind `undersize` is also emitted by `capture()` (outside the pure `classify()` above) when `save_png` itself fails on an otherwise-valid image: no PNG bytes exist, `png_bytes` stays -1, and `detail` carries the error code, so a windowed shot that cannot be written still fails red in `visual_sweep`. `capture_invalid` is warning tier: it explains a capture but never fails a scenario on its own (findings policy).

## Transport-honesty semantics

- Headless capture is impossible by engine design (RenderingServerDummy returns blank/dummy values; offscreen rendering is still open proposal #5790). Blank captures under `--headless` are engine reality, classified `transport`, never a red.
- The windowed-only skip set `WINDOWED_ONLY_SCENARIOS = {visual_sweep, visual_sweep_update}` is single-sourced in `tools/godot_dap_smoketest.py` and shared by `tools/run_playtests.py`. `display_matrix` is deliberately NOT in it: it runs under force-headless and self-skips its pixel work in-engine (`display_matrix.gd:44-47`), still emitting `display_matrix_passed {skipped: headless}` — which is also why the default run's harness-skip count is 1, not 2. (`WINDOWED_SUBPROCESS_SCENARIOS`, which does include `display_matrix`, is only the windowed-subprocess LAUNCH set used when NOT force-headless.)
- Under `PLAYTEST_FORCE_HEADLESS=1`, both harnesses report a windowed-only scenario as skipped-with-reason, never failed: the runner prints a SKIP row, the summary line reads e.g. `summary: 19/19 (1 skipped-headless)`, the summary carries `skipped_headless`, and the exit code is 0 whenever only transport skips occurred. In the default 19-scenario run only `visual_sweep` is skipped (`display_matrix` additionally self-skips in-engine with `display_matrix_passed {skipped: headless}` — a pre-existing, distinct skip; `visual_sweep_update` is not in the default set).
- `godot_dap_smoketest.py` run directly with a windowed-only scenario under `PLAYTEST_FORCE_HEADLESS=1` prints `SKIP: <scenario>: windowed-only scenario skipped under PLAYTEST_FORCE_HEADLESS (captures need a real window and renderer)`, writes an `ok: true` result file with `skipped_reason`, and exits 0.

## Duplicate-capture hook

- Triggers: environment `PLAYTEST_CAPTURE_DUPCHECK` set to anything other than `""`/`0`/`false`/`no`/`off`, OR `options.dup_check == true` (the scenario-option path). Default OFF.
- A duplicate check captures the same quiesced shot twice: a SECOND `guard_readback()` plus readback one newly rendered frame later — no game state advances between the pair (the scenario is quiesced), but the engine has rendered one more frame, and frame-to-frame readback differences in that quiesced state are exactly the nondeterminism this hook catches. Both images are converted to `Image.FORMAT_RGBA8` and their `get_data()` byte arrays compared.
- Any nonzero delta emits `capture_invalid` with `kind=nondeterministic_pair`, `classification=regression`, `luminance` from the primary, and a detail of the form `N of M pixels differ; first byte offset K` — the identified cause.
- Quarantine-tier: the primary capture still returns `ok=true`, `dup_checked` increments and feeds `visual_sweep_passed`, the primary PNG stays canonical, and the pair is compared in memory only (never saved).

## Stamp schema

`snapshot_captured` payload (Source `App.SnapshotCapture`, info tier; headless never emits it):

| Field | Type | Meaning |
| --- | --- | --- |
| `shot` | String | Filename (`09_battle.png`) or caller shot name (`battle_action`) |
| `shot_seq` | int | 1-based per SnapshotCapture instance (1..16 across a sweep) |
| `ts_msec` | int | `Time.get_ticks_msec()` at capture time — same boot clock as the record-level `ts_msec` |
| `trace_cursor` | int | Trace-log line count sampled at capture time; join key into `user://logs/agent_trace.jsonl` (this record lands at cursor+1; join with the inclusive `trace_log_has_since(event, cursor, ...)`) |
| `window` | [int, int] | `DisplayServer.window_get_size()` |
| `renderer` | String | `rendering/renderer/rendering_method` project setting (`RenderingServer.get_current_renderer()` does not exist on Godot 4.6.1 — verified; the setting resolves per platform and names the active method, e.g. `forward_plus`) |
| `godot_version` | String | `Engine.get_version_info()["string"]` (the pinned binary reports `4.6.1-stable (official)` — verified from harvested stamps) |
| `sidecar_path` | String | Canonical sidecar written next to the PNG (`res://.../<shot>.png.sidecar.json`, e.g. `res://.godot-smoke/shots/09_battle.png.sidecar.json`); `""` when the capture wrote none (`ui_render_audit`/`display_matrix` pass no metadata). The sidecar mirrors this payload's `ts_msec`/`trace_cursor` — join protocol in [snapshot-sidecar.md](../references/snapshot-sidecar.md) |

`visual_sweep_passed` payload gains (Slice 3, via `visual_sweep_baselines.gd` `report()`): `sidecar_paths` (Array[String] — the `res://.../<shot>.png.sidecar.json` path for each shot) and `invalid_captures` (int — the SnapshotCapture `invalid` counter; previously an invalid capture failed the sweep via `push_error` with no count in the payload. Structurally 0 whenever the trace fires — every invalid capture appends a `_failures` entry that stops the sweep before `report()` runs — so it doubles as a pass-time invariant; the `capture_invalid` trace carries the per-capture count). The per-region verdict fields (`region_failures`, `clusters_explained`, `clusters_unexplained`, `region_artifacts`) are NOT emitted in the trace: the engine never runs the region diff — the playtest runner records them into `.godot-smoke/playtest-report.json` (§ Explainable per-region diff, invocation-point decision).

`capture_invalid` payload (Source `App.SnapshotCapture`, warning tier): `shot` (String), `kind` (`blank`|`uniform`|`magenta`|`undersize`|`headless`|`nondeterministic_pair`), `classification` (`transport`|`regression` — magenta is always regression), `luminance` (float 0.0-1.0, 0.0 when the image is unavailable), `detail` (identified cause: byte count, magenta ratio, luma span, diff-pixel count + first offset, or fallback note).

Report stamps: `.godot-smoke/playtest-report.json` top level carries `head_sha` (`git rev-parse HEAD`), `godot_version`, `window`, and `renderer`, plus `summary` (including `skipped_headless`) and `scenarios`. The three runtime fields are harvested from the last `snapshot_captured` payload seen; absent fields are `null`, NEVER faked (headless-only runs have no `snapshot_captured` traces, so all three are null while `head_sha` is still present). Each run also deletes `.godot-smoke/result-*.json` files older than the run start so stale reds cannot contradict the report. `head_sha` is the freshness-refusal hook for `verify_all.py` (Slice 3+); `check_repo_contracts.py` verifies that, if the report exists, it carries `head_sha` and `godot_version` keys (nulls allowed). The runner also records the Slice 3 region verdict fields (`region_failures`, `clusters_explained`, `clusters_unexplained`, `region_artifacts`) on the `visual_sweep` scenario entry and files non-red drift plus unexplained clusters in a report quarantine section as kind `region_drift` — report-tier, never JSONL.

## Semantic sidecars

Every valid `visual_sweep` capture writes a canonical JSON sidecar `<shot>.png.sidecar.json` next to its PNG — the full field schema, canonical-form policy, and PNG↔sidecar↔`agent_trace.jsonl`↔`playtest-report.json` correlation protocol live in [snapshot-sidecar.md](../references/snapshot-sidecar.md). Sidecars carry the semantic render state an agent needs to explain a pixel diff with no vision: `labels` (visible labels + ink rects in stage and display px), `draw_order` (composed from `world_draw_order.gd`), `palettes` (per-region used colors), `cursor_pairs` (the known-dynamic cursor cells), `expected_regions` (the render model's ink/forbidden/string rects), `canary_rect`, `crafted_state`, `capture_env`, and `validity`.

- Canonical form: `JSON.stringify` (compact, keys sorted recursively — Godot's default, verified on 4.6.1), integer-truncated rects via `UiRenderModel.map_region`, `#rrggbb` palettes sorted lexicographically, no trailing newline. Byte-stable per seed with TWO documented run-varying fields: `ts_msec` (boot clock) drifts across runs (measured: shot 01 ts 1736 vs 1974), and `trace_cursor` (absolute line number into the append-only `agent_trace.jsonl`) shifts each run by exactly the lines the prior run appended (measured: constant delta 43 across all 16 shots between consecutive sweeps — structurally run-varying, not nondeterminism). Byte-stability verification normalizes exactly those two fields and asserts every other byte matches; each must equal the `snapshot_captured` payload's value or the trace join lies (policy single-sourced in [snapshot-sidecar.md](../references/snapshot-sidecar.md) § Canonical form). Committed baseline sidecars are written once per `visual_sweep_update` so git diffs stay clean.
- Injection split: `RenderIntrospection.collect(ctx, shot_name, crafted_state)` builds the semantic fields plus an intermediate `palette_regions` spec; `capture()` injects `shot`/`shot_seq`/`ts_msec`/`trace_cursor`/`window`/`validity` and converts `palette_regions` → `palettes` by scanning the readback image. The collector consumes the existing `smoke_context()` ctx dict — `main.gd` stays untouched.
- Label roots are selected PER SHOT KIND — `UiRenderModel.shown()` stops walking at a Viewport, so a hidden `BattleView` root does not mask labels inside its SubViewport: battle shots (09-12) collect from `BattleViewport/BattleStage` only, menu shots (06-08) from `start_menu` only, overworld shots (01-05) from nothing. Walking `battle_view` outside battle would leak stale battle HUD labels into other sidecars and poison the label join (spurious `label_moved`/`label_deleted`).
- Godot 4.6.1 API caveats (verified on the pinned binary): `Image.get_used_colors()` does not exist ⇒ palettes use a manual distinct-RGB scan (`RenderIntrospection.palette_colors`, capped to canary + hud rects — never full-frame without strided sampling); `RenderingServer.get_video_adapter_driver_info()` does not exist ⇒ `capture_env.driver_info` pinned `[]`; `get_current_renderer()` does not exist ⇒ `capture_env.renderer` reads the project setting; `get_video_adapter_name()` returns `""` under headless (harmless — sidecars are only written on valid windowed captures).
- Baseline sidecars co-copy and prune with the baseline PNGs in `visual_sweep_baselines.gd` (`_update_baselines` copies, the prune pass removes stale `*.sidecar.json`, `clear_shots` removes them), so PNG↔sidecar never desyncs. `check_repo_contracts.py` enforces region coverage: every committed baseline PNG has a sidecar sibling carrying `expected_regions` + `canary_rect` keys, and battle shots 09-12 carry a non-empty `canary_rect` and non-empty `labels`.

## Explainable per-region diff

`tools/visual_region_diff.py` (stdlib-only) joins committed baseline sidecars against fresh shot sidecars for per-region verdicts; `tools/visual_explain.py` (imported by the diff via the same importlib pattern `run_playtests.py` already uses) joins each change cluster to the sidecar for an explanation sentence. Artifacts land in `.godot-smoke/region-diff/`: `clusters.json` (machine-readable cluster bboxes + explanations), `<shot>.ascii.txt` (a 32x18 change-density grid — 1152/36 × 648/36 — an agent reads the heatmap with NO vision), and `<shot>.triptych.png` (baseline | shot | diff via the stdlib encoder `tools/png_canvas.py`). The core tools are stdlib-only — `check_repo_contracts.py` guards this; the optional vision extra stays separate.

Verdict tiers:

- RED — canary: `canary_rect` at tolerance ~0 (strict channel equality) is the FIRST RED-TIER CODED REGION. The gate compares pixels inside the rect, not the rect: a seeded 1-frame strip offset inside the DECIDUEYE canary changes pixels while the rect stays put, and fires red even when the frame delta stays under the 0.5% global gate — exactly the sub-region sensitivity the global gate is blind to.
- RED — ink/string/labels: `expected_regions` ink + string rects and baseline `labels[].display_rect` at near-zero ⇒ `region_ink_lost` / `label_deleted` / `string_drift`. Arming is evidence-based POLICY, not a code flag: these tiers are red only because the duplicate-capture noise floor measured zero across 3 consecutive seeded windowed runs (`PLAYTEST_CAPTURE_DUPCHECK=1`, zero `nondeterministic_pair` traces — evidence on file from the Slice 3 validation pass); before such evidence exists (e.g. on a new machine/driver), the tiers must stay quarantine-tier and the nondeterminism investigated.
- MASKED — known-dynamic zones: the `cursor_pairs` model-cursor cell + live-cursor rects only. The pair's `row` rect is deliberately NOT masked: it spans the whole menu row and covers label ink, so masking it blinds the string-region gate to a deleted/changed move label (this over-mask is exactly what let the seeded deleted-label perturbation slip through until caught in integration). Rows are deterministic — the crafted state pins the cursor selection — so cursor+live is the flake-safe mask. The player back-sprite lunge on `11_battle_after_attack` has no sidecar field today; if the noise-floor or flake pass shows residue there, extend the sidecar with a `dynamic_zones` field and regenerate (cheap — sidecars are git-diffable) rather than hardcoding a mask in the tool.
- BACKSTOP — global gate: the 0.5% / tolerance-8 whole-frame gate is recomputed with semantics identical to `tools/visual_diff.py` (per-channel delta strictly > tolerance) and must agree with the in-engine number. `visual_diff.py` itself — exit-code contract (0 pass / 1 drift / 2 error), global gate, and the in-engine invocation from `visual_sweep_baselines.gd` — is UNTOUCHED, so CI posture is unchanged.
- QUARANTINE — everything else: clusters outside all coded regions below the global gate become kind `region_drift` (quarantine-tier, recorded in the report quarantine section) and enter the explicit unexplained queue.

Explanation vocabulary (`visual_explain.py`): `label_moved`, `label_deleted`, `label_overlap_sprite`, `canary_absent`, `canary_shifted`, `palette_dropped`, `region_ink_lost`, plus the explicit `unexplained` tag. Every cluster gets exactly one explanation or the explicit `unexplained` tag — the unexplained queue is structural and guards false closure (zero unexplained clusters means every change is accounted for).

Invocation point: the region gate is a `tools/run_playtests.py` POST-STEP after the `visual_sweep` scenario (the in-engine `visual_diff.py` global gate runs exactly as today). The runner skips the post-step on `skipped-headless` transport and in `update`/`auto_update` mode (baselines just rewritten ⇒ trivially green); otherwise it calls `run_region_diff(shots_dir=.godot-smoke/shots, baseline_dir=docs/generated/visual-baselines, out_dir=.godot-smoke/region-diff, threshold_pct=0.5, tolerance=8)` and merges `region_failures`/`clusters_explained`/`clusters_unexplained`/`region_artifacts` into the scenario result BEFORE the row prints — non-empty `region_failures` ⇒ the row FAILs ⇒ exit 1. Standalone CLI mirrors `visual_diff.py`: `python3 tools/visual_region_diff.py --shots-dir .godot-smoke/shots --baseline-dir docs/generated/visual-baselines`, exit 0 pass / 1 drift / 2 error. The DAP transport mirrors the post-step in `godot_dap_smoketest.py` (`apply_region_step`, same verdict fields, red-tier failures flip `ok`); DAP is currently down, so the mirror is exercised only when the path is revived.

When to regenerate sidecars: the same triggers as baselines — any `project.godot` rendering/display pin change, a Godot binary bump, a driver/renderer change, or an intentional visual change — because the sidecar is the baseline's semantic twin. Regeneration rides `python3 tools/run_playtests.py --scenario visual_sweep_update`, which co-copies sidecars to `docs/generated/visual-baselines/` and prunes stale ones in the same change; PNG↔sidecar never desyncs. Region gates re-arm only after the 3-run DUPCHECK noise floor is re-measured zero.

## DECIDUEYE canary rect

The enemy canary in the battle shots is DECIDUEYE's front sprite: `front.png` is a 56x728 strip of 13 frames of 56x56, and `battle_surface_layout.gd:157` crops the first frame (`first.region = Rect2(0, 0, frame.get_width(), frame.get_width())`).

- Node rect (layout math, verified from scene + code): `EnemySprite` draws that 56x56 frame at stage rect `(96, 8, 56, 56)` inside the 160x144 `BattleViewport` (`scenes/ui/BattleView.tscn`). At the canonical 1152x648 window, `battle_view.gd` picks integer scale `floor(min((1152-32)/160, (648-32)/144)) = 4`, giving `BattleDisplay` rect `(256, 36, 640, 576)` and canary display rect `(640, 68, 224, 224)` = 50,176 px ≈ 6.72% of the 746,496-px frame. This replaces the unverified "~16x16 sprite = 0.034% of the frame" assumption cited in the plan docs (docs/superpowers/plans/2026-07-20-agent-legibility-and-vision-verification.md) by ~200x.
- Measured ink rect (verified 2026-07-20 against the committed baseline `docs/generated/visual-baselines/09_battle.png` and re-verified the same day by the windowed-verification pass against a fresh windowed capture — bit-identical to the baseline — using the `tools/visual_diff.py` stdlib decoder): frame 0 of `front.png` carries 2057 opaque pixels (all alpha 255, 7 distinct colors) with a native ink bbox of `(0, 0, 54, 56)` (2px transparent right margin). All 2057 opaque sprite pixels match the capture inside `(640, 68, 224, 224)` — 100.00% at per-channel tolerance 8, every match an exact uniform 4x4 block, confirming integer scale 4 at offset `(640, 68)`. The sprite ink therefore occupies display rect `(640, 68, 216, 224)` = 48,384 px ≈ 6.48% of the frame (node rect 50,176 px ≈ 6.72%); the ink itself is 2057×4² = 32,912 px ≈ 4.41% on screen. The battle background fills the remaining node-rect pixels, so the canary signal is the sprite ink, not whole-rect opacity. Either way the true on-screen footprint is ~6.5-6.7% — roughly 200x the unverified 0.034% assumption.
- Gate catchability: the global 0.5% gate is 3,732 px at 1152x648. A wholesale canary change (sprite swapped, blanked, or strip-loader regression) alters the 32,912 ink px = 4.41% of the frame — ~9x over the gate, comfortably CAUGHT (a whole-node-rect change would be ~13x over). The unverified 0.034% assumption (~254 px, a 16x16 sprite) sat BELOW the gate and would have been MISSED — the measurement flips the canary from "gate-blind" to "gate-visible".

## Determinism knobs (Slice 2, verify-first)

The game scales MANUALLY in two independent places — `battle_view.gd` `_layout_display()` computes an integer scale `k = floor(min((w-32)/160, (h-32)/144))` for the 160x144 battle SubViewport (Nearest-forced upsample), and the overworld camera zooms 3x over 16px tiles — with NO engine stretch (`display/window/stretch/mode` is `disabled`; `project.godot` carries no `[display]` section). The verify-first gate confirmed this before pinning anything that could fight it. Two knobs are pinned in the existing `[rendering]` section of `project.godot` (exact keys as written):

| Project setting key | Pinned value | Engine default (4.6.1, measured via `property_get_revert` on the pinned binary) | Effect |
| --- | --- | --- | --- |
| `rendering/anti_aliasing/quality/msaa_2d` | `0` | `0` (disabled) | Documents the already-disabled 2D MSAA so a future editor toggle cannot silently add per-sample blending to pixel art. Zero rendering delta by construction. |
| `rendering/2d/snap/snap_2d_transforms_to_pixel` | `true` | `false` | Quantizes 2D transforms to screen pixels at sampling time: the overworld walk lerp (`player_avatar.gd` position lerp, no camera smoothing) and any fractional camera position land on integer pixel offsets instead of smearing tile edges through the inherited Linear canvas filter. `rendering/2d/snap/snap_2d_vertices_to_pixel` stays at its default `false` (not pinned). |

`tools/determinism_verify.py pins` asserts both pins are present with these values AND that the rejected candidate pins below stay absent; it is the permanent verify-first aid (absorbed by `verify_all.py` when it lands).
